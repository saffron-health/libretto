# Unexpected UI recovery

Most workflow failures should be fixed with better selectors, waits, or assertions. Use runtime recovery when the workflow is on the right path but the site shows unexpected UI that blocks the next action.

Common examples are cookie banners, popups, modals, interstitials, and overlays that intercept clicks.

## How pageFallback works

Add `pageFallback` to a workflow when you want Libretto to recover from supported Page or Locator failures. When a wrapped method fails, Libretto calls the fallback and then retries the original method once.

```typescript
export default workflow("reviewOrder", {
  schemas: { input, output },
  pageFallback: popupRecoveryFallback({
    provider: "openai",
    apiKey: process.env.OPENAI_API_KEY!,
    model: "gpt-5.5",
    maxSteps: 3,
  }),
  handler: async ({ page }, params) => {
    await page.goto(params.url);
    await page.locator("#review").click();
    return { complete: true };
  },
});
```

## Vision recovery

The vision fallback agent looks at the current screen and takes one or more browser actions to get back to the expected path. It uses a viewport screenshot, asks the model for an action, scales the returned screenshot coordinates to Playwright coordinates, and executes the action.

`popupRecoveryFallback()` is the built-in preset for closing popups and other blockers. Use `visionRecoveryFallback()` when you want to provide your own instruction.

## When to use it

Use a fallback when exploration shows the site has nondeterministic blockers across sessions, accounts, or time. Do not add one by default, and do not use it to make business decisions that belong in the workflow handler.

Supported provider shortcut models are `gpt-5.5` and `claude-sonnet-4-6`.
