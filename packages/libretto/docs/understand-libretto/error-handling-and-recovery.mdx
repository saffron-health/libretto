# Error recovery

Libretto helps your agent handle failures in two ways: inspect failed runs so it can fix the deterministic workflow, or add a narrow runtime fallback for expected nondeterminism.

Most workflow failures should be fixed with better selectors, waits, or assertions. Runtime recovery is for cases where the workflow is on the right path but the site sometimes shows unexpected UI that blocks the next action.

## Debugging failed runs

When a workflow fails, your agent can inspect the browser state before changing code. Run headed while developing, add `pause(ctx.session)` at useful checkpoints, and use `resume` after inspection.

Libretto keeps session state, action logs, screenshots, and Playwright errors so the agent can tell whether a failure came from a selector, a timeout, navigation, or an element intercepting pointer events.

## How pageFallback works

Add `pageFallback` to a workflow when you want Libretto to recover after a Page or Locator action fails. When the action fails, Libretto calls the fallback and then retries the original action once.

```typescript
export default workflow("reviewOrder", {
  input,
  output,
  pageFallback: popupRecoveryFallback({
    provider: "openai",
    apiKey: process.env.OPENAI_API_KEY!,
    model: "gpt-5.5",
    maxSteps: 3,
  }),
  handler: async ({ page }, params) => {
    await page.goto(params.url);
    await page.locator("#review").click();
    return { complete: true };
  },
});
```

## Vision recovery

The vision fallback agent looks at the current screen and takes one or more browser actions to get back to the expected path. It is useful for nondeterministic blockers such as cookie banners, popups, modals, interstitials, and overlays that intercept clicks.

It uses a viewport screenshot, asks the model for an action, scales the returned screenshot coordinates to Playwright coordinates, and executes the action.

`popupRecoveryFallback()` is the built-in preset for closing popups and other blockers. Use `visionRecoveryFallback()` when you want to provide your own instruction.

Simple sites without nondeterministic UI usually do not need a vision fallback. For complicated sites automated through the DOM, add one when exploration shows blockers across sessions, accounts, or time. Do not use it to make business decisions that belong in the workflow handler.

Supported provider shortcut models are `gpt-5.5` and `claude-sonnet-4-6`.
