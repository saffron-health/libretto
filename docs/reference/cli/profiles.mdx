> Save and reuse authenticated browser state across runs.

### save

The `save` command saves the current browser session's cookies and localStorage as a named auth profile. Load the profile later with `--auth-profile` to skip manual login.

```bash theme={null}
npx libretto save linkedin --session app-login --site linkedin.com
```

#### When to use `save`

Only use `save` when you want to save or reuse authenticated browser state. A typical flow:

<Steps>
  <Step title="Open the site in headed mode">
    ```bash  theme={null}
    npx libretto open https://linkedin.com --headed --session app-login
    ```
  </Step>

<Step title="Log in manually">
  The browser window is visible. Log in as usual.
</Step>

  <Step title="Save the session">
    ```bash  theme={null}
    npx libretto save linkedin --session app-login --site linkedin.com
    ```

    The profile is saved to `.libretto/profiles/linkedin.json`.

  </Step>

  <Step title="Use the profile in future runs">
    ```bash  theme={null}
    npx libretto run ./integration.ts --auth-profile linkedin
    ```
  </Step>
</Steps>

#### Flags

<ParamField path="profileName" type="string" required>
  The local profile name. Passed as the first positional argument. Example:
  `linkedin`, `twitter`, or `twitter2`.
</ParamField>

<ParamField path="--session" type="string" required>
  The session to save. Use the same name you passed to `npx libretto open
  --session`.
</ParamField>

<ParamField path="--site" type="string">
  Optional site or domain label for the profile. This is documentation metadata;
  the saved profile is still referenced by `profileName`.
</ParamField>

### fetch chrome

The `profiles fetch chrome` command imports cookies and localStorage from an
existing Chrome DevTools Protocol session into a named local profile.

```bash theme={null}
npx libretto profiles fetch chrome twitter --cdp-url http://127.0.0.1:9222
```

Start Chrome with remote debugging enabled, log in normally, then run the fetch
command. The profile is saved under `.libretto/profiles/<profileName>.json`.

Recent Chrome versions reject remote debugging against the default user-data
directory; if Chrome prints that error, copy the desired profile to a temporary
user-data directory and launch Chrome from that copy.

### cloud profiles push

Push a local profile to Libretto Cloud explicitly:

```bash theme={null}
npx libretto cloud profiles push twitter --site twitter.com
```

Pushing a profile with the same name overwrites the existing cloud copy. Code
deploys are separate from profile pushes, so you can redeploy workflow code
without refreshing profile state.

### cloud credentials push

Push related `.env` credentials to Libretto Cloud by prefix:

```bash theme={null}
npx libretto cloud credentials push twitter --prefix LIBRETTO_TWITTER_
```

For example, `LIBRETTO_TWITTER_USERNAME` becomes `username` and
`LIBRETTO_TWITTER_PASSWORD` becomes `password` in the hosted credential object.
Pushing with the same credential name overwrites the cloud copy.

---

### Profile storage

Profiles are stored in `.libretto/profiles/<profileName>.json`. They are:

- Machine-local, not shared across environments or team members.
- Git-ignored by default.
- Subject to session expiry. If authentication stops working, repeat the login-and-save flow.

<Note>
  Profiles contain session cookies that are effectively credentials. The
  `.libretto/` directory is already git-ignored, but double-check your
  `.gitignore` if you're storing Libretto state in a non-standard location.
</Note>
